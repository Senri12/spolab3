### ✅ 2. Планировщик

# ❗ Чего НЕ хватает (это ключ к лабе 2)

## ❌ 1. Нет пассивного ожидания

Сейчас у тебя в pipeline:

```c
while (count >= 16) {
    ctxYield();
}
```

👉 Это **активное ожидание (busy wait)**

А нужно:

> поток **засыпает**, а не крутится

---

## ❌ 2. Нет блокировки потоков (sleep)

Тебе нужно уметь:

```c
blockCurrentThread(reason);
```

и потом:

```c
wakeupThread(threadId);
```

Сейчас такого механизма нет вообще.

---

## ❌ 3. Нет очередей ожидания

Для каждого ресурса (pipe / buffer) должны быть:

* очередь читающих
* очередь пишущих

Сейчас:

* все крутятся в while + yield ❌

---

## ❌ 4. Нет синхронизационных объектов

По заданию:

> «изучить и реализовать объекты синхронизации»

Минимум нужно:

* mutex (или хотя бы lock)
* condition / wait queue
* pipe с блокировкой

---

## ❌ 5. Нет "настоящего" stream с блокировкой

Твой `qStream` сейчас:

```c
while (count <= 0) {
    ctxYield();
}
```

А должен быть:

```c
if (count == 0) {
    sleep(readerQueue);
}
```

---

## ❌ 6. Нет группового ожидания (очень важно)

По заданию:

> **использовать групповое ожидание**

Это значит:

поток ждёт НЕ одно событие, а:

* данные появились
* или pipe закрыт
* или другой сигнал

---

# 📌 Что нужно ДОБАВИТЬ

## 1. Расширить TCB (таблицу потоков)

Добавить:

```c
state:
0 = NEW
1 = READY
2 = RUNNING
3 = FINISHED
4 = BLOCKED   // ← НОВОЕ
```

---

## 2. Сделать блокировку

Простейший вариант:

```c
void blockCurrent(int reason) {
    setTaskState(current, BLOCKED);
    saveWaitReason(reason);
    scheduleNext();
}
```

---

## 3. Очереди ожидания

Например для pipe:

```c
struct Pipe {
    buffer[16]
    count

    waitingReaders[]
    waitingWriters[]
}
```

---

## 4. wakeup

При записи:

```c
if (есть waitingReaders) {
    wakeup(reader)
}
```

При чтении:

```c
if (есть waitingWriters) {
    wakeup(writer)
}
```

---

## 5. Переделать qStream (самое главное)

Было:

```c
while (count >= 16) ctxYield();
```

Должно стать:

```c
if (count >= 16) {
    blockCurrent(WRITE_WAIT);
}
```

---

## 6. Планировщик должен игнорировать BLOCKED

В `nextRunnable()`:

```c
if (state == READY)
```

а не просто `!= FINISHED`

---

## 7. Поддержка режимов (ВАЖНО для 71)

У тебя вариант:

> **Синхронный, асинхронный**

### Синхронный:

* read/write блокируют поток

### Асинхронный:

* возвращают сразу
* если нет данных → error / skip
