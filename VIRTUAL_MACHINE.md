# TacVm13

Этот файл фиксирует текущее состояние целевой VM и соглашений генератора.

## Что используется в проекте

- Целевая архитектура: `TacVm13`
- Генерация `asm`: удаленный ANTLR-парсер через `tools/remote-parser.ps1`
- Сборка `asm -> ptptb`: `tools/remotetasks-assemble.ps1`
- Запуск `ptptb`: `tools/remotetasks-run.ps1`
- Основной ввод-вывод: hidden storages `INPUT` / `OUTPUT`
- Запасной ввод-вывод: `vm0device` из `src/TacVm13.devices.xml`

## Память

В архитектуре один основной банк:

```text
ram [0x00000000 .. 0x000FFFFF]
```

Свойства:

- размер `1 MiB`
- байтовая адресация
- little-endian
- код и данные лежат в одном `ram`

## Регистры

Основные регистры:

- `r0..r28`
- `fp = r29`
- `sp = r30`
- `r31`

Служебные storages:

- `ip`
- `cmp_result`
- `INPUT[8]`
- `OUTPUT[8]`

Важно:

- `r0` не является аппаратным нулем
- ветвления опираются на `cmp_result`
- signed-семантика для сравнений уже используется в текущем `pdsl`

## Формат инструкций

Все инструкции фиксированного размера:

```text
8 байт
```

Практически это значит:

- адреса меток в итоговом `asm` считаются в байтах
- bootstrap и функции выравниваются по шагу `8`
- генератор резолвит метки в числовые адреса до передачи в RemoteTasks

## Bootstrap

Актуальный bootstrap в начале программы выглядит так:

```asm
start:
    mov     sp, #1040000
    mov     fp, sp
    mov     r5, #180000
    mov     r6, #180004
    mov     [r5 + 0], r6
    call    main
    halt
```

Что здесь важно:

- стек стартует сверху `ram`
- слот `180000` используется как указатель кучи
- `180004` это старт heap-области
- инициализация heap теперь выполняется всегда в `start`, чтобы VM не падала на чтении неинициализированной памяти

## Стек и вызовы

Соглашение генератора:

- аргументы кладутся в стек справа налево
- результат функции возвращается в `r1`
- вызывающий код сам чистит стек после `call`

Обычный пролог:

```asm
push    fp
mov     fp, sp
sub     sp, sp, #<local_size>
```

Обычный эпилог:

```asm
add     sp, sp, #<local_size>
pop     fp
ret
```

Разметка кадра:

```text
[fp + 0]   старый fp
[fp + 4]   адрес возврата
[fp + 8]   аргумент 1
[fp + 12]  аргумент 2
...
[fp - 4]   локальная переменная
[fp - 8]   следующая локальная
...
```

## Объекты и heap

Генератор сейчас использует две разные схемы хранения:

- локальные объекты пользовательских типов получают место в кадре стека
- массивы и временные копии объектов, которые должны жить дольше одного выражения, могут выделяться в heap

Текущее поведение:

- поля объектов адресуются через `this` или через указатель на объект
- неявные поля внутри метода больше не спутываются с локальными переменными
- возврат объектов и работа с контейнерами вроде `List<Vec2i>` уже подтверждены тестом `src/classTest.txt`

## Ввод-вывод

### Основной путь: `INPUT` / `OUTPUT`

Это рекомендуемый режим запуска:

- `ExecuteBinaryWithInteractiveInput`
- `ExecuteBinaryWithInput`

Инструкции ISA:

```text
in dst   => dst = INPUT
out src  => OUTPUT = src & 0xFF
out #imm => OUTPUT = imm & 0xFF
```

### Запасной путь: `vm0device`

Файл:

- `src/TacVm13.devices.xml`

Этот путь нужен для режима:

- `ExecuteBinaryWithIo`

Он оставлен как fallback и для будущих задач.

## Где лежат `in()` и `out(...)`

Нужно различать три уровня.

### Уровень языка

В исходнике пишется обычный вызов:

```c
int c = in();
out(c);
```

### Runtime

Низкоуровневые runtime-функции лежат здесь:

- `src/runtime/in.asm`
- `src/runtime/out.asm`

Генератор дописывает эти листинги в итоговый `asm`.

### ISA

Низкоуровневые инструкции `in` / `out` определены в:

- `src/TacVm13.target.pdsl`

## Что сейчас считать источником правды

Если документация расходится с кодом, верить в таком порядке:

1. `src/TacVm13.target.pdsl`
2. `src/runtime/in.asm`
3. `src/runtime/out.asm`
4. `src/cfg_builder.c`
5. `tools/remotetasks-run.ps1`
6. итоговому `build/*.asm`
7. этому файлу
