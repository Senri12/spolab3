# TacVm13 IRQ / Scheduler Follow-Up Summary

Дата: 2026-04-04

Этот файл дополняет предыдущий лог:

- `notes/irq_investigation/2026-04-04_irq_findings.md`

Здесь собраны более поздние результаты по планировщику, памяти и `SpoInspector`.

## 1. Что подтверждено повторно

### IRQ / timer

1. Для `TacVm13` рабочий механизм ожидания прерывания это polling через:

```asm
mov [1000216], #1
mov [1000212], #1
mov r1, [1000240]
```

2. `NextInterrupt` действительно блокирует до сигнала от `Clock`.
3. После разблокировки возвращается nonzero:
   - в проверке через инспектор наблюдалось `r1 = 1`

### Планировщик

Регистровый вариант планировщика работает корректно.

Файл:

- `src/irq_sched_regs.asm`

Подтверждённый результат в отладчике:

- `r5 = 6` — прошло 6 тиков
- `r4 = 0` — round-robin корректно вернулся к task 0
- `r6 = 3` — task 0 выполнилась 3 раза
- `r7 = 30` — task 1 выполнилась 3 раза по `+10`

Это означает:

1. логика цикла планировщика корректна
2. `NextInterrupt` можно использовать как основу таймера
3. проблема `irq_sched_minimal.asm` не в самом scheduler loop

## 2. Что подтвердилось по памяти

### Базовый roundtrip работает

Файл:

- `src/mem_roundtrip_test.asm`

Проверка:

```asm
mov r0, #42
mov [200000], r0
mov r1, [200000]

mov r2, #100
mov [200004], r2
mov r3, [200004]
```

Результат:

- `r1 = 42`
- `r3 = 100`

Следовательно:

1. `mov [addr], reg` работает
2. `mov reg, [addr]` работает
3. обычные RAM store/load не сломаны сами по себе

### Store/load внутри IRQ-цикла тоже работает

Файл:

- `src/irq_mem_store_test.asm`

Подтверждённый результат:

- `r5 = 2`
- `r6 = 2`
- `r0 = 2`

Это означает, что запись в RAM внутри цикла с `mov r1, [1000240]` сама по себе тоже возможна.

## 3. Что оказалось проблемным

### `irq_sched_minimal.asm`

Файл:

- `src/irq_sched_minimal.asm`

Наблюдалось:

- `r5 = 6`
- `r4 = 0`
- но счётчики из памяти через `r2/r3` не давали ожидаемую картину

Это больше не выглядит как поломка таймера или базовой RAM.
Проблема локализуется в сочетании:

1. конкретной адресации
2. сценария отладки
3. либо особенностей доступа к близко расположенным адресам

## 4. Что найдено по `SpoInspector`

### Уже внесённая полезная доработка

`SpoInspector` теперь умеет работать с raw ASM binary без embedded debug section:

```text
[warn] Debug section 'simplelang.debug.json' not found — using empty metadata (raw ASM mode).
```

Это позволяет дебажить `.ptptb`, собранные прямо из ASM.

### Найденные баги инспектора

Были обнаружены и локально исправлены две проблемы:

1. `step` с count=1 фактически ничего не делал
2. `cont` после остановки на breakpoint не мог корректно продолжать выполнение

После правки `cont` стало возможно проходить через несколько breakpoints за один запуск скрипта.

### Ограничение всё ещё остаётся

`mem ram ...` для runtime-written данных остаётся ненадёжным способом проверки.
Для практической отладки по-прежнему лучше:

1. выгружать интересующие значения в регистры перед `halt`
2. ставить breakpoint перед `halt`
3. смотреть `regs`

## 5. Самая важная новая гипотеза

Есть сильное подозрение на aliasing / нетривиальную адресацию для близких адресов памяти.

Основания:

1. в `irq_sched_debug.asm` после 2 тиков наблюдалось:
   - `r8 = 11`
   - `r9 = 11`
   при ожидаемых `1` и `10`
2. в `addr_alias_test.asm` чтение значений с адресов `200000`, `200004`, `200008` показало не то, что ожидалось по прямой byte-address логике

Это выглядит как минимум так:

1. близкие адреса могут не вести себя как полностью независимые ячейки
2. либо адресация в `TacVm13` имеет более крупную гранулярность
3. либо проблема касается конкретного способа кодирования адреса в разных instruction forms

Важно:

Это пока **рабочая гипотеза**, а не окончательный вывод.
Её нужно отдельно добить чистым stride-тестом.

## 6. Что уже можно считать надёжным выводом

Надёжно подтверждено:

1. `NextInterrupt` работает
2. timer-based scheduler loop работает
3. round-robin логика работает
4. регистровая модель scheduler state даёт правильный результат
5. базовые load/store работают
6. `SpoInspector` в raw ASM mode пригоден для работы после локальных правок

Ненадёжно / ещё проверяется:

1. точная гранулярность RAM-адресации
2. причина странного поведения адресов `200000/200004/200008`
3. является ли это общей особенностью `TacVm13` или багом конкретных instruction forms

## 7. Практический вывод для дальнейшей работы

Следующий разумный шаг:

1. добить `addr_stride_test.asm`
2. установить минимальный безопасный шаг между ячейками данных
3. после этого перенести scheduler state / PCB на гарантированно независимые адреса
4. затем возвращаться к полноценному сохранению контекста

Если гипотеза про stride подтвердится, то PCB и служебные поля надо раскладывать не по `+4`, а по более крупному шагу.

## 8. Что дополнительно проверено по документации

Был отдельно просмотрен:

1. `build/reference`
2. `*.md`
3. `*.txt`
4. `*.pdsl`

в поиске прямых подсказок по aliasing / stride / гранулярности памяти.

### Что нашлось

Ключевые строки:

- `src/TacVm13.target.pdsl`
- `pdf_text/target-definitions (2).txt`

В `TacVm13.target.pdsl` для RAM указано:

```text
range ram [0x00000000..0x000FFFFF] {
    cell = 8;
    endianess = little-endian;
    granularity = 1;
}
```

В `target-definitions` поясняется, что `granularity` это требование к выровненности обращений к диапазону памяти.

### Вывод из этого

По спецификации:

1. `TacVm13` не должен требовать крупного шага между обычными RAM-адресами
2. гипотеза про «официальную» крупную гранулярность памяти спецификацией не подтверждается

Следовательно, если странное поведение адресов действительно есть, то его источник скорее один из следующих:

1. ошибка в текущих тестах
2. ошибка наблюдаемости через отладчик
3. баг/особенность конкретных instruction forms
4. неочевидная особенность исполнения, не отражённая в `.pdsl`

### Что не найдено

В `build/reference` и остальных текстовых файлах не найдено явного текстового объяснения:

1. aliasing близких адресов
2. stride > 4 для обычной RAM
3. специальной схемы адресации для `TacVm13`

То есть документация не даёт готового ответа на этот конкретный затык.
