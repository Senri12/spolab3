
# Текущее состояние языка и генератора TAC/VM

## Назначение документа

Этот документ фиксирует текущее поведение языка, генератора TAC и VM после серии тестов.
Нужен как памятка: какие конструкции уже работают стабильно, какие пока требуют осторожности, а какие не поддерживаются или дают неверный результат.

---

# 1. Что работает стабильно

## 1.1. Простые вызовы функций

Работают корректно обычные вызовы функций с возвратом значения.

Примеры:

```c
x = inc(5);
x = id(9);
x = add(7, 8);
```

Подтверждено тестами:

* `A=6`
* `C=15`
* `N=9`

---

## 1.2. Передача аргументов и сохранность аргументов после вызова

Работает сценарий, когда функция вызывает другую функцию и потом снова использует свой аргумент.

Пример:

```c
int useArgAfterCall(int n) {
    int t = inc(n);
    return n + t;
}
```

Подтверждено тестом:

* `B=11`

Это означает, что:

* аргументы в функцию передаются корректно,
* после вложенного вызова значение аргумента не теряется.

---

## 1.3. Рекурсия

Рекурсивные функции работают, если результат вызова сначала сохраняется в отдельную переменную.

Корректный стиль:

```c
int factSafe(int n) {
    if (n <= 1) {
        return 1;
    }
    else {
        int m = n - 1;
        int r = factSafe(m);
        int ans = n * r;
        return ans;
    }
}
```

Подтверждено тестами:

* `L=120`
* `M=0`

Значит сейчас уже работают:

* рекурсивный `call`,
* `return`,
* стек вызовов,
* базовые условия завершения рекурсии.

---

## 1.4. Условные операторы `if / else`

Простые и вложенные условия работают корректно.

Подтверждено тестами:

* `G=1`
* `H=4`
* `I=11`

Значит работают сравнения:

* `==`
* `<`
* `<=`
* `>`
* `>=`

И работают:

* одиночные `if`
* `if / else`
* вложенные `if`

---

## 1.5. Циклы `while`

Простые циклы и цикл с `if` внутри работают корректно.

Подтверждено тестами:

* `J=10`
* `K=9`

Это означает, что уже работают:

* переходы назад,
* проверка условий цикла,
* изменение переменных внутри цикла,
* вложенные условия внутри цикла.

---

## 1.6. Арифметика в простых случаях

Корректно работают:

* сложение,
* вычитание,
* умножение,
* деление,
  если выражение простое и без сложной вложенности вызовов.

Примеры:

```c
x = a + b;
x = a - b;
x = a * b;
x = a / b;
```

---

## 1.7. Композиция через промежуточные переменные

Если каждый шаг разбит на отдельную переменную, всё работает заметно стабильнее.

Пример:

```c
x = add(2, 3);
y = mul2(x);
t = inc(4);
z = y + t;
```

Подтверждено тестом:

* `O=15`

---

# 2. Что работает только при определённом стиле записи

## 2.1. Результат вызова функции лучше всегда сохранять в отдельную переменную

Безопасный стиль:

```c
int t = foo(x);
int y = a + t;
return y;
```

Небезопасный стиль:

```c
return a + foo(x);
```

На текущем этапе генератор заметно надёжнее работает именно в “трёхадресном” стиле.

---

## 2.2. Для рекурсии лучше писать развёрнуто

Предпочтительно:

```c
int m = n - 1;
int r = factSafe(m);
int ans = n * r;
return ans;
```

А не так:

```c
return n * factSafe(n - 1);
```

Хотя синтаксис может это позволять, такая запись уже приводила к проблемам при генерации.

---

## 2.3. Для сложной логики лучше использовать промежуточные переменные

Вместо:

```c
z = 10 + mul2(add(3, 4));
```

лучше писать:

```c
t1 = add(3, 4);
t2 = mul2(t1);
z = 10 + t2;
```

---

# 3. Что сейчас не работает или работает ненадёжно

## 3.1. Вложенные вызовы функций

Пока не подтверждена корректная работа выражений вида:

```c
z = inc(inc(5));
z = mul2(add(3, 4));
```

По тестам получались неверные результаты:

* `E=1` вместо `7`
* `F=0` вместо `14`

### Вывод

На текущем этапе **вызов функции внутри аргумента другого вызова считать неподдерживаемым**.

---

## 3.2. Вызов функции как часть составного выражения

Исторически уже воспроизводились ошибки на конструкциях вида:

```c
y = 10 + inc(5);
z = y + inc(4);
return n * fact(n - 1);
```

Ранее это приводило к неверному TAC, например к генерации вроде:

```asm
call 10
call y
```

То есть вместо имени функции подставлялся литерал или имя переменной.

### Вывод

Пока считать такие конструкции небезопасными.
Каждый вызов функции сначала выносить в отдельную переменную.

---

## 3.3. Повторное объявление локальных переменных в одном блоке

Такой код лучше не писать:

```c
int t = inc(5);
...
int t = inc(4);
```

Даже если парсер это пропускает, генератор может:

* перепутать слоты,
* затереть значения,
* некорректно разложить локальные переменные.

### Правило

В одном блоке использовать уникальные имена:

```c
int t1 = inc(5);
int t2 = inc(4);
```

---

## 3.4. Дублирование определений функций в одном файле

Нельзя оставлять в исходнике две версии одной и той же функции.

Плохой пример:

```c
void printInt(int x) { ... }
...
void printInt(int x) { ... }
```

Это может приводить к непредсказуемому поведению:

* парсер может брать первую версию,
* вторая может игнорироваться,
* возможны ошибки генерации.

### Правило

Каждая функция должна быть определена ровно один раз.

---

# 4. Рекомендуемый безопасный стиль программирования

На текущем этапе язык лучше использовать как “упрощённый трёхадресный код”.

## Рекомендуется

```c
t1 = add(3, 4);
t2 = mul2(t1);
t3 = 10;
z = t3 + t2;
```

```c
m = n - 1;
r = factSafe(m);
ans = n * r;
return ans;
```

```c
t = inc(5);
y = 10 + t;
```

## Не рекомендуется

```c
z = mul2(add(3, 4));
z = inc(inc(5));
return n * factSafe(n - 1);
y = 10 + inc(5);
```

# 5. Практические правила-памятка

## Можно

* простые функции,
* функции с аргументами,
* рекурсию через временные переменные,
* `if / else`,
* вложенные `if`,
* `while`,
* арифметику в простых выражениях,
* композицию через промежуточные переменные.

## Пока нельзя или нежелательно

* вложенные вызовы функций,
* вызов функции прямо внутри арифметического выражения,
* повторное объявление локальной переменной в одном блоке,
* дублирование функций в одном файле.

# 7. Текущее правило разработки

Пока генератор не доработан, писать код так, как будто язык требует явных временных переменных на каждый шаг вычисления.

То есть мыслить не как C:

```c
return a + foo(bar(x));
```

а как TAC:

```c
t1 = bar(x);
t2 = foo(t1);
t3 = a + t2;
return t3;
```

Это самый безопасный и предсказуемый стиль для текущего состояния проекта.
