# Язык и генератор

Этот файл описывает текущее состояние языка simpleLang и генератора `CFG -> TAC -> asm`.

## Что подтверждено рабочими тестами

Подтверждено на текущем проекте:

- обычные функции
- рекурсия
- `if / else`
- `while`
- массивы
- ввод-вывод через `in()` / `out(...)`
- классы
- поля и методы
- одиночное наследование
- шаблонные классы
- перегрузка функций и методов по сигнатуре
- возврат объектов из функций
- работа со специализациями вроде `List<int>` и `List<Vec2i>`

## Что теперь поддерживается в лабораторной 5

На текущем `src/classTest.txt` подтверждены:

- шаблонный класс `List<T>`
- поле-массив `T[] arr`
- методы `init`, `expand`, `add`, `printValues`
- пользовательский тип `Vec2i`
- работа со значениями `Vec2i` как с объектами
- перегруженные функции:
  - `printValue(int)`
  - `printValue(Vec2i)`
- возврат объекта из функции:
  - `Vec2i makeVec2i(int x, int y)`

То есть текущий генератор уже покрывает сценарий 5 лабы с шаблонами и перегрузкой.

## Как сейчас проходит компиляция

Цепочка такая:

```text
SimpleLang.g
-> ANTLR parse tree
-> внутреннее дерево генератора
-> CFG
-> TAC-подобный линейный код
-> asm для TacVm13
-> runtime in.asm / out.asm
```

Важно:

- генератор больше не зависит от `toStringTree()` для классовых входов
- runtime `in/out` подключается автоматически
- переходы и вызовы резолвятся в числовые адреса до RemoteTasks

## Особенности текущей модели объектов

- локальный объект получает собственное место в стеке
- ссылки на поля внутри методов корректно разрешаются через `this`
- неявные поля класса больше не объявляются как ложные локальные переменные
- для массивов и некоторых временных объектных копий используется heap
- heap инициализируется в bootstrap программы

## Ограничения, которые стоит помнить

Подтверждено и используется сейчас:

- одиночное наследование
- обычный вызов методов без отдельного механизма виртуальных таблиц
- runtime-функции `in()` и `out(...)` живут в `src/runtime`

Писать код лучше в понятном стиле:

- сложные вычисления лучше разбивать на промежуточные переменные
- для объектных программ лучше явно использовать `this.field`
- для инициализации объектов удобно использовать методы вида `init(...)`
